1. Introduction

1.1 Purpose

The purpose of this document is to describe the software requirements for a garbage and waste monitoring platform web application. The application facilitates reporting and tracking waste issues in communities, providing a user-friendly way for regular citizens to report garbage accumulation and incidents, as well as a method for authorities to manage and respond to these reports. The system also includes recycle point management functionality and interactive map-based overview capabilities. This application supports a cleaner environment and streamlines waste management efforts.

1.2 Document Conventions

The document follows IEEE section numbering standards

The document is follows the suggestions of the IEEE-SRS-Template public repository

1.3 Intended Audience and Reading Suggestions

The document's target audience is:

  • The Client (Conf. dr. Panu Andrei)
  • Current and future project managers/product owners
  • Current and future developers implementing these requirements

1.4 Product Scope

The purpose of the application is:

  • Allowing citizens to report and describe garbage accumulation incidents
  • Allowing authorities to easily manage, track and mark garbage reports as solved
  • Allowing authorities to generate PDF and CSV reports based on collected data
  • Providing interactive map-based overview with neighborhood visualization and chart analytics
  • Managing recycle points with capacity tracking and garbage collection functionality
  • Facilitating better waste management and recycling efforts
  • Provide role-based access to making, viewing and managing reports to users

2. Overall Description

2.1 Product Perspective

Garbage Monitoring for Web is a standalone web application that provides a comprehensive reporting service for end-users and integrates Leaflet mapping services, Chart.js visualization library, OpenStreetMap's nominatim geocoding API and MongoDB database for storing and managing reports. The system includes interactive map overview functionality with neighborhood polygon visualization and chart-based analytics. The reporting service supports locality-based operations and recycle point management.

2.2 Product Functions

  • Register/login for different types of users (citizen, authority, system administrator)
  • Submit location-based waste reports with description, severity evaluation and waste type
  • Mark reports as solved and manage report lifecycle
  • Interactive map overview with neighborhood polygons colored by report density
  • Chart-based analytics showing cumulative report trends by category and time
  • Generate dynamic PDF and CSV reports for specific localities and time ranges
  • Create and manage recycle points with capacity tracking and garbage collection
  • Manage and delete reports and user accounts (system administrator only)

2.3 User Classes and Characteristics

  • Citizens - Basic users, can report issues
  • Authorities - Local authorities, can report issues and generate county-specific reports
  • Application Administrators - Developers, administrators, can view and delete reports and user accounts

2.4 Operating Environment

  • Platform: Web browser
  • Backend: Node.js
  • Database: MongoDB

2.5 Design and Implementation Constraints

  • No usage of frameworks
  • Open-source libraries only
  • Usage of AJAX
  • Web services are exposed through a RESTful API

2.6 User Documentation

  • Setup + installation guide for local deployment
  • Descriptive video presentation of application functionality

2.7 Assumptions and Dependencies

  • Users have internet access
  • Access to a browser with JavaScript enabled
  • OpenStreetMap Tiling API is available
  • OpenStreetMap Nominatim API is available
  • Database provided in the .env file is available

4. System Features

4.1 User Authentication and Role Management

4.1.1 Description and Priority: High priority. The system must provide secure user registration, login, and role-based access control supporting three user types: Citizens, Authorities, and Administrators.

4.1.2 Stimulus/Response Sequences:

  • User accesses signup page → System shows sign-up page
  • User submits registration data → System validates, hashes password, creates account, and redirects to appropriate dashboard
  • User attempts login → System validates credentials and creates session
  • User accesses protected resource → System verifies session and role permissions
  • User clicks logout → System destroys session and redirects to home page

4.1.3 Functional Requirements:

  • REQ-1.1: Allows user registration with username, email, password, and role selection
  • REQ-1.2: Hashes passwords using bcrypt before storage
  • REQ-1.3: Validates unique email and username combinations
  • REQ-1.4: Creates and manages user sessions with time-based expiration
  • REQ-1.5: Enforces role-based access control for all protected routes
  • REQ-1.6: Provides user validation and role change functionality for administrators

4.2 Interactive Garbage Report Creation and Management

4.2.1 Description and Priority: High priority. All users must be able to create detailed garbage reports by selecting locations on an interactive map and providing descriptive information. Authorities must be able to mark reports as solved.

4.2.2 Stimulus/Response Sequences:

  • User clicks location on map → System displays marker and fills coordinates in the form
  • User fills form (category, severity, description) → System validates input fields
  • User submits report → System performs reverse geocoding, stores report in database → User receives confirmation message
  • Authority views report → Authority marks report as solved → System updates report status

4.2.3 Functional Requirements:

  • REQ-2.1: Displays interactive map
  • REQ-2.2: Allows users to place markers by clicking on map locations
  • REQ-2.3: Automatically determine locality and county via Nominatim reverse geocoding
  • REQ-2.4: Provides severity rating, incident description and waste type selection
  • REQ-2.5: Allows authorities to mark reports as solved

4.3 Interactive Map Overview and Analytics

4.3.1 Description and Priority: High priority. The system must provide an interactive map overview showing neighborhoods as colored polygons based on report density, with integrated chart analytics for data visualization.

4.3.2 Stimulus/Response Sequences:

  • User accesses overview page → System displays map with neighborhood polygons colored by report density
  • User selects locality from dropdown → System loads corresponding GeoJSON and updates map visualization
  • User configures chart parameters → System generates cumulative report chart by category and time range

4.3.3 Functional Requirements:

  • REQ-3.1: Displays interactive map with neighborhood polygon overlays
  • REQ-3.2: Colors neighborhoods based on report density calculations
  • REQ-3.3: Provides locality selection dropdown with supported locations
  • REQ-3.4: Generates cumulative report charts using Chart.js with date range filtering
  • REQ-3.5: Displays chart analytics for different waste categories over time

4.4 Data Export and Report Generation

4.4.1 Description and Priority: Medium priority. Authorities must be able to generate PDF and CSV reports filtered by locality and date ranges for administrative purposes.

4.4.2 Stimulus/Response Sequences:

  • Authority user accesses dashboard → System displays locality selection, date filters and generation buttons
  • User chooses export format (PDF/CSV) → System queries database and aggregates data
  • System generates requested report → System provides download link

4.4.3 Functional Requirements:

  • REQ-4.1: Generates PDF and CSV reports with grouped data by locality
  • REQ-4.2: Supports date and locality filtering
  • REQ-4.3: Reports include statistical analysis and aggregated information

4.5 Recycle Point Management System

4.5.1 Description and Priority: Medium priority. Authorities must be able to create and manage recycle points with capacity tracking and garbage collection functionality.

4.5.2 Stimulus/Response Sequences:

  • Authority creates recycle point → System stores location, capacity and contact information
  • Authority adds garbage to recycle point → System updates fill amounts and validates capacity limits
  • Authority clears recycle point → System resets fill amounts to zero

4.5.3 Functional Requirements:

  • REQ-5.1: Allows creation of recycle points with location, capacity and operational details
  • REQ-5.2: Tracks fill amounts by waste category with capacity validation
  • REQ-5.3: Provides garbage addition and clearing functionality
  • REQ-5.4: Manages recycle point operational hours and contact information

4.6 Administrative Dashboard System

4.6.1 Description and Priority: Medium priority. Administrators must have tools to manage user accounts and reports.

4.6.2 Stimulus/Response Sequences:

  • Admin accesses dashboard → System displays user management and report management tables
  • Admin clicks delete user → System prompts confirmation and removes user account
  • Admin clicks delete report → System prompts confirmation and removes report

4.6.3 Functional Requirements:

  • REQ-6.1: Displays table of all reports with filtering capabilities
  • REQ-6.2: Displays table of all users with role management
  • REQ-6.3: Provides delete functionality for users and reports
  • REQ-6.4: Shows system statistics and counts

5. Other Nonfunctional Requirements

5.1 Safety Requirements

  • Misleading reports can be deleted by the admin

5.2 Security Requirements

  • Database security - using parametrized queries through MongoDB
  • Passwords are hashed before being stored
  • Session management with bearer token authentication
  • XSS (Cross Site Scripting) protection through input sanitization
  • RESTful API with proper HTTP cache control headers

5.3 Software Quality Attributes

  • Usability: intuitive interface and simple UI
  • Safety: gracefully handle errors and exceptions
  • Scalability: modular codebase with strong emphasis on the Model View Controller pattern

5.4 Business Rules

  • Only authorities can generate reports and mark reports as solved
  • Only authorities can create and manage recycle points
  • Only admins can delete user accounts
  • Only admins can delete reports and validate users
  • Users can only view and delete their own reports
  • Recycle point capacity limits must be enforced when adding garbage
