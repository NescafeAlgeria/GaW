1. Introduction

1.1 Purpose

The purpose of this document is to describe the software requirements for a garbage and waste monitoring platform web application. The application facilitates reporting and tracking waste issues in communities all around the world, providing a user-friendly way for regular citizens to report garbage accumulation and incidents, as well as a method for autorities to manage and respond to these reports in their respective counties. This application supports a cleaner environment and streamlines recycling efforts.

1.2 Document Conventions

The document follows IEEE section numbering standards

The document is follows the suggestions of the IEEE-SRS-Template public repository

1.3 Intended Audience and Reading Suggestions

The document's target audience is:

  • The Client (Conf. dr. Panu Andrei)
  • Current and future project managers/product owners
  • Current and future developers implementing these requirements

1.4 Product Scope

The purpose of the application is:

  • Allowing citizens to report describe and garbage accumulation
  • Allowing authorities to easily manage and track garbage accumulation reports
  • Allowing authorities to generate reports based on the data collected from user reports
  • Facilitating better waste management and recycling efforts
  • Provide role-based access to making, viewing and managing the reports to users

1.5 References

  • IEEE SRS Template
  • W3C Web Standards
  • MongoDB Documentation
  • Leaflet JavaScript Library
  • MVC Design pattern (core pattern behind application)

2. Overall Description

2.1 Product Perspective

Garbage Monitoring for Web is a standalone web application that provides a reporting service for the end-users and integrates Leaflet mapping services, OpenStreetMap's nominatim geocoding API and MongoDB database for storing and managing reports. The reporting service is not limited to a specific region, rather allowing people all around the world to use it.

2.2 Product Functions

  • Register/login for different types of users (citizen, authority, system administrator)
  • Submit location-based waste reports with description, severity evaluation and waste type
  • Generate dynamic PDF and CSV reports for a specific county and time range
  • Manage and delete reports and user accounts (system administrator only)

2.3 User Classes and Characteristics

  • Citizens - Basic users, can report issues
  • Authorities - Local authorities, can report issues and generate county-specific reports
  • Application Administrators - Developers, administrators, can view and delete reports and user accounts

2.4 Operating Environment

  • Platform: Web browser
  • Backend: Node.js
  • Database: MongoDB

2.5 Design and Implementation Constraints

  • No usage of frameworks
  • Open-source libraries only
  • Usage of AJAX
  • Web services are exposed through a RESTful API

2.6 User Documentation

  • Setup + installation guide for local deployment
  • Descriptive video presentation of application functionality

2.7 Assumptions and Dependencies

  • Users have internet access
  • Access to a browser with JavaScript enabled
  • OpenStreetMap Tiling API is available
  • OpenStreetMap Nominatim API is available
  • Database provided in the .env file is available

4. System Features

4.1 User Authentication and Role Management

4.1.1 Description and Priority: High priority. The system must provide secure user registration, login, and role-based access control supporting three user types: Citizens, Authorities, and Administrators.

4.1.2 Stimulus/Response Sequences:

  • User accesses signup page → System shows sign-up page
  • User submits registration data → System validates, hashes password, creates account, and redirects to appropriate dashboard
  • User attempts login → System validates credentials and creates session
  • User accesses protected resource → System verifies session and role permissions
  • User clicks logout → System destroys session and redirects to home page

4.1.3 Functional Requirements:

  • REQ-1.1: Allows user registration with username, email, password, and role selection
  • REQ-1.2: Hashes passwords using bcrypt before storage
  • REQ-1.3: Validates unique email and username combinations
  • REQ-1.4: Creates and manages user sessions with time-based expiration
  • REQ-1.5: Enforces role-based access control for all protected routes

4.2 Interactive Garbage Report Creation

4.2.1 Description and Priority: High priority. All users must be able to create detailed garbage reports by selecting locations on an interactive map and providing descriptive information.

4.2.2 Stimulus/Response Sequences:

  • User clicks location on map → System displays marker and fills coordinates in the form
  • User fills form (category, severity, description) → System validates input fields
  • User submits report → System performs reverse geocoding, stores report in database → User receives confirmation message

4.2.3 Functional Requirements:

  • REQ-2.1: Displays interactive map
  • REQ-2.2: Allows users to place markers by clicking on map locations
  • REQ-2.3: Automatically determine locality and county via Nominatim reverse geocoding
  • REQ-2.4: Provides severity rating, incident description and waste type selection

4.3 Data Export and Report Generation

4.3.1 Description and Priority: Medium priority. Authorities must be able to generate PDF and CSV reports filtered by county and date ranges for Administrative purposes.

4.3.2 Stimulus/Response Sequences:

  • Authority user accesses dashboard → System displays county selection, date filters and generation buttons
  • User chooses export format (PDF/CSV) → System queries database and aggregates data
  • System generates requested report → System provides download link

4.3.3 Functional Requirements:

  • REQ-3.1: Generates PDF and CSV reports with grouped severity mapping
  • REQ-3.2: Supports date and county filtering
  • REQ-3.3: Reports include the cleanest and dirtiest localities

4.4 Administrative Dashboard System

4.4.1 Description and Priority: Medium priority. Administrators must have tools to manage user accounts and reports.

4.4.2 Stimulus/Response Sequences:

  • Admin accesses dashboard → System displays user management and report management tables
  • Admin clicks delete user → System prompts confirmation and removes user account
  • Admin clicks delete report → System prompts confirmation and removes report

4.4.3 Functional Requirements:

  • REQ-4.1: Displays table of all reports
  • REQ-4.2: Displays table of all users
  • REQ-4.3: Provides delete button for users
  • REQ-4.4: Provides delete button for reports

5. Other Nonfunctional Requirements

5.1 Safety Requirements

  • Misleading reports can be deleted by the admin

5.2 Security Requirements

  • Database security - using parametrized queries through MongoDB
  • Passwords are hashed before being stored
  • Session management
  • XSS (Cross Site Scripting) protection

5.3 Software Quality Attributes

  • Usability: intuitive interface and simple UI
  • Safety: gracefully handle errors and exceptions
  • Scalability: modular codebase with strong emphasis on the Model View Controller pattern

5.4 Business Rules

  • Only authorities can generate reports
  • Only admins can delete user accounts
  • Only admins can delete reports
